# 雙北 YouBike 場站車輛配置模擬器

## 專案目標
分析雙北 YouBike 各場站的車輛配置策略，計算每站「6:00 配置」與「15:00 配置」的最佳車輛數，並判定每站屬於最優解、次優解或最後解。產出一個互動式 HTML 模擬器（可部署到 GitHub Pages）與 Excel 報表。

---

## 核心資料與符號定義

### 借減還（最重要，務必理解符號）
- 「借減還」= 借車數 − 還車數
- **正值 = 借 > 還 = 民眾把車騎走 = 場站車輛「減少」**
- **負值 = 還 > 借 = 民眾把車還回 = 場站車輛「增加」**
- 庫存遞推公式：`stock[h+1] = stock[h] − 借減還[h]`（正值讓庫存減少）
- 這個符號慣例對「民眾借減還」與「調度借減還」**完全一致**：兩者正值都代表場站車輛減少。

### 三份原始資料（皆為 5/18~5/22、期別1+2）
1. **民眾借還**（場站流量檔）：每站每小時的民眾借減還。**這是求解的唯一依據。**
2. **調度借還**（調度流量檔）：每站每小時的調度借減還。**僅作圖層參考，不納入求解。**與民眾借還互不包含、各自獨立。
3. **歷史綁車**：每站每小時的綁車數。**目前僅作場站資訊參數（平均最大綁車數），不納入求解、不畫圖。**

### 資料前處理
- 同站同日「期別1 + 期別2」相加，再對 5 天取平均。
- **保守取整**：正數往 +∞（ceil）、負數往 −∞（floor），即往絕對值大的方向取整。
- 綁車的「平均最大綁車數」：每站每日的最大綁車數，再取 5 天平均。
- 車位數缺失的站補預設值 24。
- 所有數值最終皆為整數。

## 求解邏輯

### 模型
- 設場站總車位數為 C，每個整點 h（6~23 時）的庫存為 stock[h]。
- **A = 6:00 配置**（上午段起始庫存）：上午段 6~14 時的庫存從 A 開始，依民眾借減還遞推。
- **B = 15:00 配置**（下午段重設庫存）：15:00 重設為 B，下午段 15~23 時依民眾借減還遞推。
- 上午段（A）與下午段（B）獨立求解，因為 B 只影響下午段。

### 判定（標準判定，目前 v7~v10 採用）
- 對所有整點 h ∈ [6, 23]，要求 `0 < stock[h] < C`（庫存須大於 0 且小於總車位數）。

### 分級
- **最優解**：存在 (A, B) 使全部 18 個整點時刻皆合格。
- **次優解**：上午段、下午段各最多 1 個時刻違反。
- **最後解**：其餘（借還量超出車位容量，兩段配置無法完全滿足）。
- **車位過少(≤1)**：車位數 ≤ 1 的站，無法求解。

### 目前分級結果（5/18~22 民眾借還，標準判定，共 3285 站）
- 最優解：2,153 站
- 次優解：121 站
- 最後解：1,011 站

## 多次調度與自動求解

模擬器允許在任意整點新增「調度點」（補車/調出），不限於 6:00 和 15:00。

### 「🪄 自動求最少調度」演算法
- 用貪婪法：從 6:00 開始，找最遠能延伸到幾點仍不違反（區間波動 ≤ C−2），到不行就在該處放一個調度點，重複到 23:00。
- 實測對「最後解」場站若允許多次調度，多數可達最優解（需 2~5 次調度）。

### ⚠️ 已知問題（待修）
1. **自動求解在極端站會給出錯誤方案**：曾發現「綁車緩衝」被錯誤地放寬區間波動上限（把 bindBuffer 加進 limit），導致演算法給出「庫存爆到車位 1.5 倍卻自認 OK」的假方案。修正方向：演算法應「猜 + 用真正的 isViolating() 驗證 + 修正」迭代，而非用近似公式一次算完不驗證。（註：v7 之後綁車已退出求解，此 bug 影響降低，但自動求解仍應加上「驗證後才回傳」的保險。）
2. **部分場站資料疑似異常**：如「松仁路95巷口」上午淨借出達 461 台（車位僅 74），「捷運三民高中站(2號出口)」、「捷運淡水站」（綁車達 902）等。這類站單日借/還流量遠超車位數，本質上兩段配置無法滿足。可能是期別1+2相加放大了數值，需評估是否設流量上限或標示為異常。

## HTML 模擬器規格（目前 v10）

### 介面結構（單一畫面，避免滾動）
- 左側：場站清單（可收合 ◀▶）
- 右側：詳細頁（場站資訊、配置控制、每小時格子、圖表、摘要卡片）

### 篩選器（5 個，可組合）
場站搜尋、城市、行政區、責任區、分級。已選條件顯示可拆除標籤，行政區選單依城市動態收窄。

### 場站基本資訊
場站代號、城市、行政區、責任區徽章、總車位數、**平均最大綁車數徽章**。

### 每小時格子
- 大數字 = 該時刻在站車輛數（庫存）
- 時間標籤用單一數字（6、7、8... 23，不用 06:00）
- 紫色框 = 調度點，可輸入「調入(+)/調出(−)」量（顯示對場站的影響量，非絕對庫存值）
- ★ = 6:00 / 15:00 主要配置點（不可刪除）
- 紅底 = 違反（庫存 ≤0 或 ≥車位）
- 點 + 新增調度、點 − 移除調度

### 圖表（每小時曲線，6~23 時）
- **可勾選圖例**，5 條線各自獨立開關：
  - 民眾行為（對場站）：綠虛線，預設**開**
  - 調度行為（對場站）：紫虛線，預設**開**
  - 在站車輛數（庫存軌跡）：綠實線，預設**關**
  - 車位上限：紅虛線，預設開
  - 0 線：橙線，預設開
- **「對場站的影響」畫法**：民眾/調度曲線皆畫「借減還取負號」，使「往上 = 對場站車輛增加、往下 = 減少」成為全圖統一語言。例如調度拉車出站 → 線往下。
- X、Y 軸刻度皆為**整數**。Y 軸範圍只納入「目前有顯示」的線，上下界各往外加 10。
- tooltip 顯示目前有開的線的數值（民眾、調度兩條都開時加顯示「合計影響」）。

### 技術注意事項
- 單一 HTML 檔，所有場站資料以 JSON 內嵌（約 1MB）。
- Chart.js 透過 CDN 載入（`cdn.jsdelivr.net/npm/chart.js@4.4.0`）。
- **不可使用 localStorage / sessionStorage**（部分環境不支援）；狀態存記憶體即可。
- 每個場站的使用者修改獨立保存在記憶體 `state` 物件，切換場站不遺失，但重整頁面會還原。

## 資料來源（TDX，若要接即時資料）
- 站點靜態：`https://tdx.transportdata.tw/api/basic/v2/Bike/Station/City/{Taipei|NewTaipei}`（含總車位數 BikesCapacity）
- 即時車位：`https://tdx.transportdata.tw/api/basic/v2/Bike/Availability/City/{Taipei|NewTaipei}`（含 AvailableRentBikes 在站車輛數）
- 認證：OAuth2 Client Credentials（token URL: `https://tdx.transportdata.tw/auth/realms/TDXConnect/protocol/openid-connect/token`）
- **金鑰放環境變數 `TDX_CLIENT_ID` / `TDX_CLIENT_SECRET`，切勿寫進程式碼或提交到 git。**
- ⚠️ TDX 即時 API 只給「當下快照」，**沒有每小時借減還流量**。要自建歷史流量需定時抓快照、相鄰時間相減反推。
- 頻率限制：每 IP 每分鐘最多 20 次。

### v11 即時資料架構 (GitHub Actions + JSON 歸檔)
- `scripts/fetch_tdx.py`：抓雙北即時車位，存成 `data/{日期}/{小時}.json`，並維護 `data/index.json`(日期->小時清單)。
- `.github/workflows/fetch.yml`：每小時整點(cron `0 * * * *`)觸發，跑抓取腳本後 commit 回 repo。也可手動 workflow_dispatch。
- 金鑰放 GitHub Secrets：`TDX_CLIENT_ID` / `TDX_CLIENT_SECRET`。
- 前端 index.html 用 fetch 讀 `data/index.json`、`data/{日期}/{小時}.json`：
  - 即時線：讀最後一天最後一小時的快照，畫成水平線(代表「目前」在站數)。
  - 回放線：選定日期後載入該日全部小時，畫成當日在站變化曲線。
  - 每小時 setInterval 自動重抓最新。
- **資料對應問題待注意**：TDX 站碼是 StationUID(如 YouBike2.0 格式)，與本專案 code(場站代號)可能不同。前端目前用 `endsWith/includes` 模糊比對，若對不上需建立 StationUID <-> 場站代號 對照表(可從 TDX Station API 的 StationID 欄位建立)。
- TDX 給的是在站車輛數(存量)，與歷史的借減還(流量)是不同的量，圖上分開呈現。

## 開發慣例
- 版本命名：HTML 檔名加 _vN，每次改動遞增（v11、v12...）。
- 純視覺/介面調整不改求解結果；只有「換資料期間、改取整方式、改判定邏輯、改是否納入調度/綁車」才會改變分級。
- 每次改 HTML 後，同步更新 GitHub Pages 部署用的 index.html。
- 求解相關改動務必驗證：先用幾個已知場站（如捷運科技大樓站、松仁路95巷口）核對庫存軌跡與分級。
